# ADR 0001 — Abstraction layer: acausal thermal-fluid component network (MTK)

**Status:** Proposed (Body confirmation requested before first build)
**Date:** 2026-05-21
**Deciders:** Body + Soul Council (Architect leading; Skeptic, Accountant, Advocate consulted)

This ADR satisfies the Soul's AL gate — *"Before any implementation: name the
abstraction layer explicitly. Write it down."* No Julia is written before it.

---

## Context

The project (see `CONTEXT.md`) builds a fuel-channel→coolant→loop model in
Julia/ModelingToolkit, with two goals: a Soul-System process dogfood, AND a
**technical benchmark against frozen netflow** on performance, complexity, cost,
accuracy, and capability (extensibility toward subchannel / partial / full core).

**netflow's abstraction (the "before"):** node = one scalar state; edge = a
conserved flux `f(s_a, s_b)`; the residual is flux-conservation at interior nodes,
**hand-assembled** into a `scipy.sparse` system and solved by damped Newton (Picard
fallback). Its thermal plugin built a `FuelRod → ForcedConvection → CoolantAdvection`
chain; it scaled to ~380k nodes (~190 s) and a 17×17×30 assembly (~8 s, ~7 Newton
iters), verified exactly vs the textbook fuel-pin closed form and code-compared to
VERA P6/P7. *(All netflow figures here are self-reported from its README/docs;
they must be re-measured by re-running netflow on this machine before any benchmark
claim — see CONTEXT.md §5.)*

The Council's pivot verdict: netflow's abstraction *is* the Modelica/MTK acausal
across/through connector. So the MTK formulation expresses the **same physics**, but
conservation is **generated structurally** by `connect()` instead of hand-assembled.
**That contrast is the benchmark.**

## Decision — the abstraction layer

> **A network of acausal thermal-fluid components joined by typed across/through
> connectors, where conservation is generated by ModelingToolkit's connection
> semantics (not hand-assembled). Discretization — axial nodes, radial nodes,
> channel count — is parametric, so the same component definitions scale from one
> fuel pin to a multi-channel assembly. Steady-state first (algebraic, no storage);
> storage terms are added later for transient WITHOUT changing the connector
> contracts.**

### Connectors

- **HeatPort** (reuse, MTKStdLib): across `T` [K], through `Q_flow` [W].
- **FluidPort** (custom — MTKStdLib has none; its Hydraulic is isothermal): through
  `ṁ` [kg/s], transported specific enthalpy `h` (or `T` for the constant-cp first
  cut); `p` [Pa] carried when momentum is added (not in the first cut).

### Components (steady-state first cut)

- **Fuel radial path:** power source (`q'·L`) → fuel conduction → gap → clad
  conduction → clad-outer `HeatPort`. Composed from MTKStdLib `ThermalResistor` +
  a heat-flow source (or one custom `@mtkmodel`).
- **ConvectiveCoupling:** clad-outer `HeatPort` ↔ coolant, via `ConvectiveConductor`
  (G = h·A; h from Dittus-Boelter/Gnielinski, mirroring netflow).
- **CoolantVolume (custom):** inlet/outlet `FluidPort` + a `HeatPort`; steady energy
  balance `ṁ·(h_out − h_in) = Q_in`; single-phase properties (constant cp first,
  property package — Clapeyron/CoolProp/SteamTables — later).
- **Boundaries:** inlet (`ṁ`, `T_in`), power BC; outlet open.

## What varies / what decides / what cannot vary (First Principle)

**What varies (instantiation knobs):** `N_axial`, `N_radial`, `N_channels`; power
profile (uniform | cosine); property model (constant-cp | IF97); steady | dynamic.

**What decides whether it varies:**
- *Capability goal* → `N_axial`/`N_channels` MUST be parameters generated
  programmatically (array-of-components) from day 1 — the extensibility-toward-core
  benchmark depends on it.
- *Accuracy goal* → `N_radial`, property fidelity.
- *Fidelity phase* → storage terms (steady omits; dynamic adds `HeatCapacitor` +
  fluid accumulation), connectors unchanged across the transition.

**What cannot vary without breaking everything (the verification invariants):**
1. **Across/through contract** — at a connection, across equal, through sums to zero
   (enforced by `connect()`).
2. **Energy conservation** — Σ deposited power = Σ coolant enthalpy rise (steady),
   to machine precision. The primary verification invariant (netflow's
   "flux sums to zero at nodes," generalized).
3. **Mass conservation** — `ṁ` continuity along the channel (steady).
4. **Upwind enthalpy** — coolant carries its own enthalpy downstream.

## Consequences

- We do **not** hand-assemble residuals/Jacobians — MTK does. This is the central
  difference from netflow and what the benchmark measures.
- We do **not** reinvent steam/water properties (reuse a package; constant-cp first).
- **Verification:** the same textbook fuel-pin closed form netflow used (exact).
  **Code comparison:** the same VERA P6/P7 path, AND netflow itself as a second code
  — code-to-code, never called validation ([[validation-vs-code-comparison]]).
- **Benchmark axes operationalized:** performance (solve time vs N), complexity
  (LOC / component count for the same physics), cost (dev effort), accuracy (vs
  closed form + code comparison), capability (effort to grow `N_channels` / add a
  phenomenon).

## Out of scope

A component *library* ("Julia TRANSFORM"); full-plant; two-phase properties;
measured-data validation. (Re-confirmed with the Body, CONTEXT.md §5.)
