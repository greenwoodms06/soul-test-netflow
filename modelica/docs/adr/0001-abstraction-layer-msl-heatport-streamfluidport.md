# ADR-0001 — Abstraction layer: MSL HeatPort + stream FluidPort + IF97 medium

**Status:** accepted
**Date:** 2026-05-26
**Decision-maker:** Body (greenwoodms06@gmail.com), via grilling 2026-05-26
**Supersedes:** none
**Related:** soultest-julia ADR-0001 (the same-shape decision in the MTK ecosystem)

## Context

This project is the third leg of a paradigm triangulation (netflow Python → MTK Julia → Modelica). The First Principle of the Soul System requires the abstraction layer to be **named explicitly and written down** before any implementation.

Modelica is an acausal modelling language. Connectors carry both an *across* variable (potential — temperature, pressure) and a *through* variable (flow — heat rate, mass-flow). This shape differs from netflow's single-scalar Node + single-scalar flux Edge. Therefore the abstraction the `.mo` package commits to is *not* netflow's — it is Modelica's native acausal connector formalism. The netflow Node/Edge surface is preserved only as a narrative comparison in CONTEXT.md / FINDINGS.md.

## Decision

**Abstraction layer for `NetflowModelica`:**

> A network of acausal thermal-fluid components joined by *typed Modelica Standard Library connectors* — `Modelica.Thermal.HeatTransfer.Interfaces.HeatPort_a/_b` (T / Q_flow) for conduction and `Modelica.Fluid.Interfaces.FluidPort_a/_b` (p / m_flow / stream h_outflow & Xi_outflow) for convective transport — with conservation generated by Modelica's `connect()` rather than hand-assembled. Coolant properties come from `Modelica.Media.Water.StandardWater` (IAPWS-IF97). Discretisation (axial slices, radial layers) is parametric so the same components scale from one pin to many; steady solves use Dymola's nonlinear solver, transients use the default integrator (Dassl/Cvode/IDA at Dymola's choice) with `HeatCapacitor` storage at thermal nodes.

### What this commits us to

1. **Components have *typed* connectors, not single-scalar ports.** A fuel-pin component exposes `HeatPort_a clad_outer` (T, Q_flow). A coolant cell exposes `FluidPort_a port_a` and `FluidPort_b port_b`. A convection edge has one `HeatPort_a` and one `FluidPort_a/_b` pair.
2. **Conservation is generated, not hand-assembled.** A Modelica `connect(rod.clad_outer, conv.solid)` emits the Kirchhoff equations automatically (sum-of-flows = 0; potentials equal). Unlike netflow's manual flux-summation residual, the user never writes this.
3. **Stream connectors for fluid energy transport.** `FluidPort` carries `h_outflow` as a *stream* variable — the value the connector contributes when flow leaves the component. Bi-directional flow is handled by `inStream(port_a.h_outflow)` for upwind enthalpy. This is the Franke et al. 2009 design pattern that the Julia attempt re-implemented from scratch in MTK; here we use the MSL implementation directly.
4. **Coolant medium = `Modelica.Media.Water.StandardWater`.** IAPWS-IF97 full medium. cp, ρ, μ, k, Pr come from `Medium.specificHeatCapacityCp(state)` etc., not from constant parameters. (The Julia attempt used constant cp = 5500 J/kg/K because no IF97 MTK binding ships ready.)
5. **Steady-state-first.** Slices 1–4 are time-invariant. Storage (`HeatCapacitor`) is added in slice 5 without changing connectors — Modelica supports adding storage at nodes without altering surrounding component contracts.
6. **Same physics closures as netflow.** UO₂ k(T) Fink/Petri, Zircaloy-4 k(T), gap conduction + optional gray radiation, Dittus-Boelter h(Re,Pr), analytic 1/(4π k L) pellet centerline-to-surface resistance. These are implemented as Modelica `function`s and called from inside components — the same shape netflow uses for Materials and Fluid closures.

### Mapping (for the reader who knows netflow)

| netflow | NetflowModelica |
|---|---|
| `Node` (scalar T) | a temperature variable inside a component; exposed via `HeatPort_b heat` for external coupling |
| `Node` (scalar h, coolant) | a specific-enthalpy variable inside `CoolantCell`; exposed via stream `FluidPort_a/_b port_a/_b` |
| `Edge.flux(s_a, s_b)` | the component's internal equations; `connect()` handles conservation |
| `CylindricalConduction` | a `function` returning conductance × ΔT, inside the FuelPin component's equations |
| `ForcedConvection` | a `Convection` component with `solid` HeatPort + `fluid` FluidPort |
| `CoolantAdvection` (upwind) | `inStream(port_a.h_outflow)` in the cell's energy balance — the upwind is handled by the stream-connector semantics |
| `FuelRod.q_lin` source | `PrescribedHeatFlow` at the centerline node, or a parameter inside the FuelPin internal equations |
| `Network.solve_steady` | Dymola's nonlinear solver invoked by `simulateModel(..., stopTime=0)` or a static experiment |
| `Network.solve_transient` (BDF) | Dymola's integrator (Dassl default) on the DAE form |

## Alternatives considered

- **Mimic netflow's Node/Edge with a generic scalar connector.** Rejected by grilling Q2: forces Modelica into an unidiomatic shape; loses the connect()-emitted conservation; confuses anyone who reads the package as Modelica.
- **Build two parallel packages (netflow-mimic + idiomatic), prove they agree.** Rejected by grilling Q2: doubles the work; the comparison axis is well-enough served by the FINDINGS narrative.
- **TRANSFORM as the build base.** Rejected by grilling Q4: TRANSFORM equipment-style components would be a use-vs-build comparison; this leg keeps the apples-to-apples with Julia's MTKStdLib choice.
- **Constant-cp coolant for Julia parity.** Rejected by grilling Q7: IF97 is the one capability where this leg outperforms Julia out of the box; using it is the dogfood-relevant choice.

## Consequences

**Positive:**
- Anyone fluent in Modelica can read `NetflowModelica` as Modelica, not as a port.
- Conservation is connect()-generated, removing a class of bugs netflow had to validate against by hand.
- The IF97 medium is "free" — one parameter swap if we want constant-cp later.
- The component contracts (HeatPort / FluidPort) are the *same* surface TRANSFORM and ThermoPower use, so the package is composable with mature ecosystems if a future scope opens that door.

**Negative:**
- Component code is heavier than netflow's plugin (more boilerplate per component: variables, parameters, equations block, typed connectors).
- Debugging connect()-emitted equations is opaquer than reading netflow's explicit residual.
- The cost-of-IF97 vs constant-cp is non-zero: each property evaluation does a state-function call. We accept this; if it dominates we add a parameter to fall back.

**Open questions / deferred:**
- Whether Dymola's default integrator handles the pellet/clad/coolant stiffness without manual tuning. Re-evaluated at slice 5.
- Whether the IF97 medium triggers the "negative-pressure warning" event-storm that the dymola-sim-debug skill names. Re-evaluated at slice 2 (the first slice with a real medium).

## How we will know this was right

- Slice 4 numerical match vs netflow within <100 mK (Julia's bar was 25 mK).
- Slice 3 + 4 LOC count comparable to or less than the Julia ThermalChain.jl module (98 lines).
- No need to hand-write a single conservation equation outside a component's own internal balances.
- A reader new to the codebase can locate "where does heat get from clad to coolant?" by following `connect()` statements, not by reading a global residual assembler.
